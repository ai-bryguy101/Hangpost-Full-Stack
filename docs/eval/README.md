# Recommendation ranker — offline evaluation

This is the measurement half of "closing the ML loop" (CLAUDE.md §5).
The app logs every recommendation it surfaces
(`recommendation_impressions`) and every action a user takes on it
(`recommendation_outcomes`). This harness turns that log into a score:
**is the ranker actually better than dumb baselines?**

## What it measures

For each **query** (one `/recommendations` call — a set of impressions
sharing a `source_user_id` + `surfaced_at`), we derive a graded
relevance label from each impression's outcome:

| Outcome | Relevance |
|---|---|
| `friend_request_sent` or `hangout_rsvped` | 3 |
| `profile_opened` | 2 |
| `viewed` | 1 |
| nothing | 0 |

We then score three orderings of each query and average across queries:

- **`live_ranker`** — the order the API actually surfaced (by `rank_position`).
- **`popularity`** — globally most-surfaced candidates first (a strong, dumb baseline).
- **`random`** — a seeded shuffle (the floor).

with two metrics:

- **NDCG@10** — rewards putting high-relevance items near the top.
- **Recall@10** — what fraction of all relevant items made the top 10.

The ranker earns its keep only if it beats popularity, which beats
random. The math lives in
[`hangpost_api/recommendations/evaluation.py`](../../apps/api/src/hangpost_api/recommendations/evaluation.py)
(pure stdlib, unit-tested in `apps/api/tests/test_evaluation.py`), so it
can be reused by the Phase 7 retraining harness without a database.

## Generating a real report (Codespaces)

The operator works browser-only (CLAUDE.md §1), so run this in the
Codespace terminal against the live database:

```bash
docker compose -f infra/compose/docker-compose.yml exec api \
  python -m scripts.evaluate --days 7
```

It writes `docs/eval/<date>-eval.md` and prints the table. Capture some
outcomes first by clicking through `/demo` (view a few picks, open a
couple, hit "Add friend") — with zero outcomes every relevance is 0 and
the report says so honestly.
